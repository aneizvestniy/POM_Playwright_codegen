# 02 — Page Object Model (POM)

## Мета модуля

Навчитись будувати автотести так, щоб вони:
- легко читались (тест = сценарій)
- були дешевими в підтримці
- не ламались від дрібних UI змін

## Коли POM потрібен

POM має сенс, коли:
- є повторювані сценарії (login, search, checkout)
- UI часто змінюється
- у проєкті більше 10–20 UI тестів

## Ключова ідея

**Тести не мають знати про HTML.**

- Page Object / Component містить локатори та методи дій.
- Тест викликає методи рівня бізнес-дії: `login()`, `addGuest()`, `openSettings()`.

## Рекомендовані правила

1) **Селектори живуть в PO/Components**, а не в тесті.
2) В PO — **actions**, у тесті — **assertions** (зазвичай).
3) Не роби "god page" (все в одному класі) — дроби на компоненти.
4) Назви методів як бізнес-дії.

## Типова структура

- `src/pages/` або `src/pageObjects/`
- `src/components/`
- `tests/`
